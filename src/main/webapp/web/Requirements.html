1 Introduction

1.1  Aim

To develop a system which allows users to register for an online account where they will be able to create diary events with other registered users. System on creation of events should show if invited guests are available and  show a grid calendar layout of events.

1.2  Key personnel

Development team

  • Greg Lindert

1.3  Rationale for development

The new system will be developed using Java Enterprise Edition (J2EE) and JavaServer Faces (JSF) technologies.

1.4  Timetable of release

  • First release (M1 mandatory requirements)
  • Tuesday 26th March 2019
  • Second release (M2 mandatory requirements)
  • Monday 6th May 2019

2 Requirements Table

Type

1

The system must store following details about a user:

  • their username (comprising at least 6 alphanumeric characters), which must be present and must be unique among all active users
  • their password, which must never be displayed by the system
  • their firstname(s) and lastname separately
  • their address, including a valid postcode if it is in the UK
  • their phone number, which, if present, can validly be in any one of the formats illustrated below (where spaces are not significant):
  • +44 (23) 92846438
  • (023) 92846438
  • their email address, which must be present and in a valid format

M1

2

A user must be able to create a new user account by providing the information above.

M1

3

Only valid records shall be added to the database.

  • If the user enters an invalid record, the system indicates to the user where invalid data has been entered and provides them with an opportunity to correct it.

M1

4

User can only alter their information once they are logged in

M1

5

System must provide a shared searchable online contact list facility to any logged in user

M1

6

Validation of records must be done on entry/amendment.

M1

7

The application must also provide an online diary facility.

M2

8

A diary appointment consists of:

  • a date/time at which the appointment starts
  • a date/time at which the appointment finishes
  • a textual name of the appointment
  • a textual  description of the appointment
  • the person who created the appointment (the "owner")
  • a set of people who are involved in that appointment (which may or may not include the owner)

M2

9

No appointment can clash with any other appointment for any of the people involved. Creation of any such appointment must not be allowed. A possible error report might allow the creator to alter the set of people involved.

M2

10

The owner of an appointment must be able to "cancel" it when shown in any of the reports. Cancelling an appointment will make all of the people involved in it "free".

M2

11

A "calendar" view of appointments

M2
