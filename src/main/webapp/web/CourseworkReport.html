Online Diary Web Application

Introduction

Implementing Online Diary Web Application will help me to deepen my knowledge and understanding of Java EE development and its techniques. I aim to use Persistence Techniques to create and access data, as well as create an easy to use application to any user, that will persist only validate data at all stages of the application.

Design

For the design of this web application, I have focused on enhancing the user experience by making it user-friendly. I have tried to plan and develop this application to be simple and easy to use for any user.

Key Components

Account Login and Registration

Most of today's applications require a feature that can store and retrieve personal details within an account. Therefore I have designed this application having this in mind. Diagram 1 shows Part of Flow Diagram. It sets out the steps the user needs to take in order to create an account and sign in. The user is always redirected to the main register and sign in screen.

Diagram 1 - Part of Flow Diagram of Login and Registration process

If any of the fields are missing data or any of the data is not valid, an error message is displayed below the incorrect input to the user. This is demonstrated in two examples in Image 1. Only once all inputs are filled in and validated the user is registered and redirected to the login page where can log in.

Image 1 - Registration Form validation and Error message

I have chosen this design and process as it’s easy to use for users, error messages are clear, easy to see therefore easy to correct, and the system is secure with the password hashing and the fact that the password never is decoded, also the password is never displayed.

Profile view and edit

A key part of this application is the ability to view and update personal details. The user can view a profile and then on the same page with a press of a button he/she can edit the personal information. Diagram 2 shows the detailed flow of this process, as it can be seen from the flow diagram, there is a step to validate the inputted data. As all of the validation that was done on registration is again done on saving of updated details so that the data consistency and correctness are kept.

To make the application maintainable I have created a personal information form template that is used on the registration page as well as the profile page. This way the information on both pages are the same and in future is any changes are needed to be made to either page the change will happen in both places. The template consists of two parts a view mode part where the inputs are hidden and labels are shown and an edit mode where input is shown and enabled and labels are hidden.

Diagram 2 - Part of Flow Diagram Profile view/edit and Contacts Search

Contacts search

Another key aspect of the application is a contact search that can only be accessed by logging in. Diagram 2 also shows the process following a successful login in. Although the diagram shows that the first and last name is needed for the search to happen this has been changed since. I have changed the design to make it simpler to search for a contact, now there is only one input field and that input field is used to search for similarities in all personal fields. Another change that isn't displayed on the flow diagram is that an ajax event is used for searching, as the user types a database query is done and the table with contacts is updated.  

Creation and editing of Events

Diary entries or events is the main component and feature of this web application. I have designed this part of the application again to be simple and user-friendly. I have split the event creation into two parts, event details and event guests invitations.

Creation of event consists of giving the event a title, description, picking a start and end date and time, as well as searching and inviting guests to the event. Searching for guests uses the same technique and template as the contacts page, as the user types to search for users, the records are filtered and displayed to the user according to the text that was typed.  

Diagram 3 - Part of Flow Diagram showing Creating and viewing of Events

Throughout implementation some changes have been made to the design, Diagram 3 shows the flow of the creation of events, it shows a busy search view part, this has been changed and has not been implemented to constrained time and complexity of the project. However, a different approach to showing if guests are busy was taken.

A major part of inviting guests is checking if a guest is available on this day and time. This is done by searching through the events table to see if any invited guest has an event or is a guest of an event, during the time selected by the user that is creating the event. If any of the invited guests are busy during this time, an overall error message and an icon is displayed next to busy guest as illustrated below in Image 2.

Image 2 - Event Creation Form  

Calendar view of events

The last main aspect of the application is a calendar view with all the events. Once a user signs in, the application displays an agenda of today's events. User can then click on an event to view it, cancel it, or edit it. The user is only privileged to deleting and editing of the events if said logged in user is the owner of the event. A user can also click on an empty time slot in the day view to create a new event, the clicked time slot is sued as the default start time and a default one hour later finish time.

The user has also the ability to view events on a large calendar view, on this page the user can see future and past events, and perform the same actions of cancelling, viewing and editing an event. The view can be also changed to week and day. On a month view, a user can click on a day and create an event on that day, with the default start time of now and finish time as one hour later.

Implementation and Testing

Implementation

Front End

The front end of the web application was mainly developed using JSF with help from Primefaces Framework. I have decided to use Primefaces as it is simple to use and has a great performance, as well as it is one of the most used frameworks when developing in Java EE. It’s easy to start and quick to get up and running, most of the complex components are created and ready to use. For example, the calendar is one of the most complex components that I was able to use, with little experience of Primefaces. It also styles all elements the same using one theme which makes the application look professional and seamless throughout.

Back end

Back end was built using standard Java EE principles, using EJB and JPA. I have created three entity classes which are linked with each other, Person, Address and Event. Together with a controller and service for each class. I have also created several other controllers that I used throughout the application, for example, I have created and used a controller for searching for contacts. It has a view scope to not store data for too long and only keep the data for this view.  

Security

To secure this application, I have opted in to use Application managed security and as part of the mechanism, I have set up and used a servlet filter, to block the request and response from passing any further if a user is not signed in. To do this I created a class that implemented from Filter class, I have then overwritten the doFilter() method to check if there is an active HTTP Session, if there was an active session then the user continued as normal, if a user was not logged in and tried to access a secured page the would be redirected to login page.

I have also used decided to use password hashing. On registration the password is hashed using SHA-256 encryption and stored in database, on login, the password entered by user is also hashed using the same technique and then compared against the stored password in the database that matches the username, only if both username and password match the user that is logged in, and an HTTP session is created. On log out, the session is destroyed and data from it is deleted, and the user is redirected to the login page.

Problems

The main problems I have faced were with Netbeans, the IDE environment as well as Glassfish the web server that has been used to run the application. The problem occurs to be that the server was unreliable and kept crashing, after a small change to either front end or back end a full rebuild and deployment was needed, and sometimes the server crashed so badly that a full restart was needed. I have tried to use a different IDE and Server like Wildfly but that has bought other problems along the way, so I had to revert. There wasn't a real fix to this problem, the way I have managed to deal with this problem was to do a run every time you changed something, I have also tried to do several changes at once to minimise the waiting time, and from time to time a Clean and Build helped, to have a clean and fresh build.

Another major issue I have faced is with retrieving data from database, when I have loaded the user and its events the data was incomplete, after some research I have discovered that my one-to-many relationships and many-to-many relationships have a loading type of lazy which means that data is only retrieved when needed, after changing the loading type to Eager it worked perfectly.

Testing

Testing was carried out throughout the development of the project, as a new feature was implemented all the previously implemented features where tested. To test all implemented features I have created several accounts, this allowed me to test the registration and login in process. With each account, I have tried to input incorrect data and tried to proceed. I have implemented a sign-in feature early at the beginning of the project, this meant that each time the server was restarted I had to re-login, however, by doing this, I have thoroughly tested this aspect, by logging in many many times.

To test events and availability of users, I have started by creating several events using one person and inviting some users writing down the event details and guests invited, afterwards I have logged in as one of the not invited guest and tried to create an event at the same time with the same guests and seeing if it allowed me to do so. This was the trickiest part and required several iterations and fixes.

Summary

To conclude by designing and implementing Online Diary Web Application, I have learned how to develop Enterprise Level applications, it has shown me the strong sides and weak sides of these kinds of applications. The strong side being reliability and stability, one the application is running its a solid piece of software that is hard to break if implemented correctly. The main weak side is the development process, it’s sometimes long and irritating when after each change to code the server needs to be restarted, this means that implementation time is longer.  

For the future work, this application has a few places it can improve, one such place is more features could be implemented, for example, a forget or change password feature would be very useful, due to time constraint and complexity of project this I was unable to implement this feature. Another feature that I have planned but did not implement, is a graphical way of showing which users are busy and free on a given day, this would make the checking for users availability much simpler and more user-friendly.

One of the good decision that I have made during the development of this application was using Primefaces as the framework for frontend development, as it was very easy to use and understand and most of the big component have been created and could be used as tag which saved time, and as the components have been used by many  people they have been tested and improved over time. A bad design decision I have made is splitting the personal details and address details into two classes, it bought more complexity into the project in the beginning, it did have a reason, potential and more  features could be implemented by doing it this way, such as creating reports on residences living at  an address or picking an address from database if exists, meaning  less duplicate records. However, those features never have been developed but would be beneficial for implementing in the future.
